Frequently asked questions

Answers about how Memact works, what apps can see, and how consent is controlled.

Basics

What is Memact?v

Memact helps apps understand users' digital activity, using only the activity and memory permissions they approve.

Where does Memact run?v

Apps use a small Memact client SDK. Evidence can come from the browser extension and, if enabled, a local helper. Memact turns approved evidence into context, schemas, and memory; apps do not receive a blanket activity feed.

Does an app get my whole memory graph?v

No. Apps only get the context, summaries, evidence, or graph objects allowed by the permissions and categories you approve. Raw graph access is separate and sensitive.

How does consent work?v

It shows what an app wants Memact to understand before anything is connected. You approve or cancel. Apps should link Data Transparency beside consent so you can review the evidence, schemas, memory objects, and context they plan to use.

What are activity categories?v

They limit which part of your activity Memact can understand for that app, like research pages, news, AI conversations, developer activity, or media.

Connect instructions

How do I connect an app to Memact?v
  1. Register your app in the Memact portal and choose the smallest scopes and activity categories your feature needs.
  2. Add a Connect Memact button in your app that sends users to /connect?app_id=...&scopes=...&categories=...&redirect_uri=....
  3. Place a Data Transparency link beside the consent flow. Explain the evidence fields, context objects, summaries, graph packets, retention, and revocation path.
  4. After the user approves, Memact redirects back to your redirect_uri with a connection_id. Store that id for the signed-in user in your app.
  5. Keep the raw Memact API key in server environment config, such as MEMACT_API_KEY in .env locally and a secret manager in production.
  6. Your backend calls Memact's verification endpoint with Authorization: Bearer <api_key>, the stored connection_id, required scopes, and activity categories.
  7. Only request the approved context, memory, schemas, or graph objects returned by Memact. Do not ask for more than the user consented to.
What does the verification endpoint return?v

When verification succeeds, Memact returns allowed: true along with the effective scopes, activity categories, an understanding strategy that tells your app which category-specific evidence Memact can use, and the user's connection id. If verification fails, it returns allowed: false with an error code explaining whether the API key is invalid, consent is missing, or scopes/categories are denied.

What is the verification endpoint URL?v

Use POST https://api.memact.com/v1/access/verify with Content-Type: application/json and Authorization: Bearer mka_.... The request body should include connection_id, required_scopes, and activity_categories. Your app backend calls this endpoint; never call it from a browser bundle or mobile app.

Developers

How do I use the Memact API in an app?v
  1. Register your app in Memact and choose the smallest scopes and activity categories your feature needs.
  2. Add a Connect Memact button in your app. Send users to `/connect` with app id, scopes, categories, and redirect URL.
  3. Put a Data Transparency link beside that consent flow. Explain the evidence fields, context objects, summaries, graph packets, retention, and revocation path.
  4. After approval, Memact redirects back with a `connection_id`. Store that id for the signed-in user in your app.
  5. Keep the raw Memact API key in server environment config, such as `MEMACT_API_KEY` in `.env` locally and a secret manager in production.
  6. Your backend calls Memact's verification endpoint with the API key, stored connection id, required scopes, and activity categories.
  7. Request only the approved context, memory, schemas, or graph objects returned by Memact.
Where should the Memact API key live in code?v

Treat the raw Memact API key like a server-side secret. This is the private app key that starts with `mka_` and is created in the Memact portal. Do not hard-code it into browser bundles, mobile apps, public repos, logs, shared prompts, or user-facing settings panels.

Do I need a Supabase public key?v

No. App developers configure only the private Memact app key on their own backend. Supabase keys belong to Memact's access infrastructure, not customer apps.

Is a Data Transparency page required?v

Yes. Any app asking users to consent through Memact should expose Data Transparency next to the consent flow. It must explain the evidence fields, context objects, summaries, graph packets, retention, and revocation path. Categories alone are not enough.

What is not allowed?v

Apps should only request what they need. Hidden monitoring, selling raw activity or memory, manipulation, and sensitive decision-making are not allowed.
